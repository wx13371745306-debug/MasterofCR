# 炒菜菜谱扩展指南

本文说明在项目中**新增炒菜菜谱**（以及需要时新增可炒食材）时应操作的步骤与注意事项。炒菜逻辑由 `FryRecipeDatabase`、`FryPot`、`FryableItem` 与 `FryIngredientDefinition` 协作完成。

---

## 一、先理解：配方是怎么匹配的

- 玩家往锅里放的每一种食材，会对应一个 **`FryIngredientDefinition`（食材 SO）** 引用，锅内按「食材种类 → 数量」累计。
- 成功匹配某条菜谱的条件是：
  - 该菜谱在 `FryRecipeDatabase` 的列表里，且 **`unlocked` 为 true**；
  - 锅内**恰好**包含菜谱里列出的每一种食材，且**各自数量与配置一致**；
  - 锅内**不能多也不能少**：食材种类数量必须与该菜谱的 `ingredients` 条目数一致（锅里有额外种类则不会命中）。
- 若没有任何菜谱匹配，会使用 **`failedDishRecipe`（失败兜底）** 的配置，而不是「无结果」。

因此：新菜谱里写的食材组合，必须和玩家实际能扔进锅里的 **`FryIngredientDefinition` 引用**一一对应。

---

## 二、仅增加新菜谱（使用已有食材）

适用：新菜只用当前项目里已经存在的 `FryIngredientDefinition`（例如已有番茄、蛋等资产）。

1. 在 Project 窗口中找到 **`FryRecipeDatabase`** 资源（默认使用 [`Assets/OS/recipe.asset`](../Assets/OS/recipe.asset)）。
2. 在 Inspector 中展开 **Recipes**，点击 **+** 增加一条 `FryRecipe`。
3. 填写新菜谱字段（常用项）：
   - **recipeName**：唯一且稳定，便于脚本或订单用名字查找（与 `FindByName` 一致）。
   - **unlocked**：是否已解锁；未勾选则不会被匹配。
   - **ingredients**：为每种食材添加一行 **IngredientEntry**：
     - **ingredient**：拖入对应的 **`FryIngredientDefinition`** 资产；
     - **count**：该食材需要几份（通常为 1）。
   - **resultPrefab**：盛出后生成在手上的成品预制体（通常带 `CarryableItem` 等）。
   - **finishedVisualPrefab**：烹饪结束、仍留在锅内的成品模型。
   - **eatenPrefab**：吃完后的空盘等（若暂无可填 `None`）。
   - **订单与结算**：**size**（桌面槽位）、**dishIcon**、**price**、**eatTime** 按需求填写。
4. 保存资源（`Ctrl+S` / 自动保存视项目设置而定）。
5. 确认场景里的 **炒锅（`FryPot`）** 上 **`recipeDatabase`** 引用仍指向该 `FryRecipeDatabase`（一般无需改，除非你换了资源）。

---

## 三、增加新食材 + 新菜谱

适用：新菜需要一种新的可炒原料（例如新蔬菜）。

### 1. 新建食材定义（SO）

1. 菜单 **Assets → Create → Cooking → Fry Ingredient**（或右键 Create 同级菜单）。
2. 建议把资产放在统一目录（例如 `Assets/OS/FryIngredients/`），命名清晰，如 `MyVegetable.asset`。
3. 填写：
   - **id**：字符串唯一标识；若该食材会在商店出售，建议与 [`ShopItemCatalog`](../Assets/OS/ShopItemCatalog.asset) 里对应条目的 **`itemId` 保持一致**，便于对照与排查。
   - **displayName**：展示用名称（可选；可与 `id` 相同）。
   - **icon**：可选，用于 UI。

### 2. 世界中的食材预制体（可被抓取、可下锅）

1. 复制或新建食材 **Prefab**，挂上 **`FryableItem`**：
   - **ingredient**：拖入上一步的 **`FryIngredientDefinition`**；
   - **baseRequired / addedRequired**：影响该食材对烹饪进度条的贡献（第一份与后续份可不同）；
   - **visualInPotPrefab**：放入锅中后锅内显示的小模型（通常无 Collider）。
2. 确保该 Prefab 同时具备与现有食材一致的交互组件（如 **`CarryableItem`**、刚体、碰撞体等），与项目内其它食材预制体保持一致。

### 3. 商店（若需要购买该食材）

在 **`ShopItemCatalog`** 中新增一条 **ShopItemEntry**：

- **itemId**：与食材 SO 的 **id** 对齐（推荐一致）；
- **worldPrefab**：上述食材预制体；
- 其余：价格、图标、是否解锁等按需填写。

### 4. 在菜谱中使用新食材

按 [第二节](#二仅增加新菜谱使用已有食材) 在 **`FryRecipeDatabase`** 里为新菜配置 **ingredients**，把新 **`FryIngredientDefinition`** 拖进列表并设置 **count**。

## 四、检查清单（避免常见错误）

| 检查项 | 说明 |
|--------|------|
| 食材引用一致 | 菜谱里的 `FryIngredientDefinition` 与食材 Prefab 上 `FryableItem.ingredient` 必须是**同一资产**。 |
| 种类与数量 | 锅内食材种类集合必须与该菜谱 **完全一致**；多一种或少一种都不会匹配。 |
| 失败兜底 | 未匹配时会走 **failedDishRecipe**；请确认该兜底已配置且预制体可用。 |
| 炒锅引用 | 场景中 **`FryPot.recipeDatabase`** 指向包含新菜谱的 **`FryRecipeDatabase`**。 |
| 订单/结算 | 若订单系统按 **`recipeName`** 或其它字段绑定，新增菜谱后需在对应系统里一并配置（若有）。 |

## 六、小结

- **只加新菜**：在 **`FryRecipeDatabase`**（如 `recipe.asset`）里加一条 `FryRecipe`，配好 **ingredients** 与成品/视觉/订单字段即可。
- **加新食材**：新建 **`FryIngredientDefinition`** → 在食材 Prefab 的 **`FryableItem`** 上绑定 → 可选在 **`ShopItemCatalog`** 上架 → 再在菜谱里引用该 SO。

按上述流程操作，**无需再改枚举或改脚本**即可完成大部分新菜部署。
